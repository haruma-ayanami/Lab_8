Лабораторно-практическая работа №8

Поведенческие паттерны проектирования: Посредник и Состояние

Выполнил: Ткачев А. Д.
Группа: 2407СА1
Проверил: Макаров М. С.

Цель работы

Изучить поведенческие паттерны проектирования «Банды четырёх (GoF)» на примере разработки приложения, моделирующего систему управления очередью печати документов. Освоить на практике паттерны Посредник (Mediator) для организации взаимодействия между компонентами системы и Состояние (State) для управления поведением объектов в зависимости от их текущего состояния. В процессе выполнения работы необходимо научиться проектировать архитектуру приложения с низкой связанностью компонентов, обеспечивать гибкость управления состояниями объектов и реализовывать возможность расширения функциональности системы без изменения существующего кода.

Задание

Разработать консольное приложение, моделирующее работу системы управления очередью печати документов.
Программа должна обеспечивать:
Реализацию механизма изменения поведения документа в зависимости от его состояния с использованием паттерна State. Документ может находиться в следующих состояниях: новый документ, документ в процессе печати, успешно напечатанный документ и документ с ошибкой печати.
Реализацию централизованного взаимодействия между компонентами системы с использованием паттерна Mediator. Взаимодействие между документом, принтером, очередью печати, логгером и диспетчером должно осуществляться исключительно через посредника без прямых ссылок друг на друга.
Поддержку очереди печати документов по принципу FIFO. Документы должны добавляться в очередь и обрабатываться последовательно.
Имитацию работы принтера с возможностью возникновения ошибки во время печати документа и последующего восстановления системы.
Логирование ключевых событий работы системы, таких как добавление документа в очередь, начало печати, успешное завершение печати и возникновение ошибки.
Демонстрацию работы программы в основной функции приложения с выполнением нескольких сценариев: успешная печать документа, обработка ошибки печати и повторная отправка документа в очередь после восстановления.

Теоретическое обоснование

Поведенческие паттерны проектирования предназначены для организации взаимодействия между объектами системы и распределения ответственности между ними. Их использование позволяет уменьшить связанность компонентов программы, повысить гибкость архитектуры приложения и обеспечить возможность расширения функциональности без изменения существующего кода.
Паттерн Состояние (State) используется для изменения поведения объекта в зависимости от его текущего состояния. В этом случае объект изменяет свою реакцию на одни и те же действия в зависимости от внутреннего состояния. Каждое состояние представляется отдельным классом, реализующим общий интерфейс поведения. Контекст содержит ссылку на текущее состояние и делегирует выполнение операций соответствующему объекту состояния. В рамках данной лабораторной работы документ выступает в роли контекста, а состояния нового документа, печати, успешного завершения и ошибки представлены отдельными классами состояний.
Паттерн Посредник (Mediator) предназначен для организации взаимодействия между объектами через специальный объект-посредник. Вместо прямого взаимодействия друг с другом компоненты системы обмениваются сообщениями через посредника. Такой подход позволяет уменьшить связанность между объектами системы и централизовать управление взаимодействием между ними. В данной лабораторной работе посредник управляет взаимодействием между документом, принтером, очередью печати, логгером и диспетчером.
Использование данных паттернов позволяет построить гибкую архитектуру приложения, обеспечить управление состояниями объектов и централизовать взаимодействие компонентов системы без прямых зависимостей между ними.

Описание выполненных действий

Для выполнения лабораторной работы было разработано консольное приложение, моделирующее систему управления очередью печати документов.
На первом этапе был реализован паттерн State. Для этого был создан класс Document, выполняющий роль контекста конечного автомата состояний. В данном классе хранится ссылка на текущее состояние документа, а выполнение операций делегируется объектам состояний через интерфейс IDocumentState. Были реализованы конкретные состояния документа: NewState, PrintingState, DoneState и ErrorState. Каждое состояние определяет допустимые действия и переходы между состояниями документа.
На втором этапе был реализован паттерн Mediator. Для этого был создан интерфейс IMediator, определяющий общий способ взаимодействия компонентов системы через посредника. Далее был реализован класс PrintSystemMediator, выполняющий координацию взаимодействия между документом, принтером, очередью печати, логгером и диспетчером. Все компоненты системы взаимодействуют друг с другом исключительно через посредника без прямых ссылок между собой.
Также были реализованы классы Printer, PrintQueue, Logger и Dispatcher, выступающие в роли коллег посредника. Принтер выполняет имитацию печати документа и может генерировать событие ошибки. Очередь печати обеспечивает последовательную обработку документов. Логгер фиксирует ключевые события работы системы. Диспетчер управляет добавлением документов в очередь и запуском обработки очереди печати.
В основной программе была продемонстрирована работа реализованных паттернов. Были созданы несколько документов, добавленных в очередь печати. Далее была выполнена последовательная обработка документов, включая сценарий успешной печати, сценарий возникновения ошибки печати и повторную отправку документа в очередь после восстановления состояния.

Результат выполненной работы

В результате выполнения лабораторной работы было разработано консольное приложение, моделирующее систему управления очередью печати документов.
Проверка работы программы показала, что использование паттерна State позволяет изменять поведение документа в зависимости от его текущего состояния без использования условных операторов в классе контекста. Переходы между состояниями реализованы в отдельных классах состояний, что обеспечивает удобство расширения системы и добавления новых состояний документа.
Использование паттерна Mediator позволило централизовать управление взаимодействием между компонентами системы и устранить прямые зависимости между ними. Все компоненты системы взаимодействуют через посредника, что упрощает сопровождение программы и расширение её функциональности.
Совместное применение паттернов State и Mediator позволило повысить гибкость архитектуры приложения, уменьшить связанность компонентов системы и обеспечить возможность дальнейшего расширения функциональности без изменения существующего кода.
